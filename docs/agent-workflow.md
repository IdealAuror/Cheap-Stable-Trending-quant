# Agent 工作流程：epic-leek-quant

## 总则

```
高级模型 → research-spec → Flash → 代码 + 结果
                                        ↓ 代码审核（独立步骤）
                                        ↓
                                 高级模型 → 分析结论 + 更新沉淀
```

---

## 第一步：研究方向定义（高级模型）

**输入：** 待研究的问题

**高级模型输出——一份 `research-spec`：**

```yaml
假设: "EV<0 因子在 A 股具有显著的正向选股能力"
失败条件: "IC_IR < 0.2 或 Q1 多头组未能跑赢基准"

回测设置:
  时间区间: 2014-01 至 2026-06
  股票池: 沪深300成分股（剔除 ST、上市不满 180 天）
  基准: 000300.XSHG 全收益
  换仓频率: 月频（每月第一个交易日）
  数据版本: 使用聚宽 get_fundamentals(date=) 的 PIT 模式，不可使用期末值

因子定义:
  字段名: factor_ev_negative
  公式: EV = market_cap + total_liability - cash_equivalents
  条件: EV < 0 → 1，其他 → 0
  字段映射:
    market_cap: valuation.market_cap
    total_liability: balance.total_liability
    cash_equivalents: balance.cash_equivalents

数据处理:
  异常值: 对 market_cap / net_profit 做 1% 和 99% winsorize
  缺失值: 剔除关键数据缺失（market_cap/pe_ttm/net_profit 任一为空）
  中性化: 按申万一级行业做行业中性化，不做市值中性化

分析方法:
  主指标: Spearman Rank IC（月度截面）
  辅助: Q1-Q5 分组月收益（等权），仅报告 Q1 多头组绝对收益
  显著性: Newey-West t 统计量（滞后 4 期）
  通过标准: Rank IC_IR > 0.3，且 Q1 多头组年化收益 > 基准年化收益 + 2%
  补充: 分市值档（大/中/小）和分行业 IC

输出要求:
  表格1: 月度 Rank IC 序列描述统计（均值、标准差、NW-t、IC_IR、胜率、正负比例）
  表格2: Q1-Q5 分组年化收益、年化波动、最大回撤、夏普比率
  表格3: 分市值档 Rank IC
  额外: Q1 多头组累计净值曲线时间点数据（每月末净值），用作画图
```

---

## 第二步：具体执行（Flash 模型）

**输入：** 第一步的 `research-spec`

**工作方式：**
- 按 spec 编写执行代码
- 在聚宽平台上运行，产出结果
- spec 中有歧义时：**先标注歧义点，按最保守的方式处理，明确写在偏差记录中**
- 严禁"跳过"——所有问题都必须被记录，不做静默忽略

**产出物：**
```
1. 执行代码（完整可运行的聚宽策略文件）
2. 执行结果：
   - spec 中要求的全部表格
   - 净值曲线数据点（CSV 格式）
   - 日志/警告信息（如果有）
3. 执行偏差记录：
   - 每处歧义：spec 怎么写 vs 你实际怎么处理的
   - 数据缺失：spec 要求的字段中哪些聚宽没有、如何处理
   - 执行异常：代码运行中的报错或异常情况
```

---

## 第三步：代码审核（高级模型 或 独立审查）

**位置：** Step 2 完成之后、结果分析之前

**必须在看结果之前审核代码。** 这是防止未来函数和确认偏误的关键门禁。

**审核清单：**
- 数据调用是否使用了 PIT 模式？(审查 `get_fundamentals` 的 date 参数)
- 未来函数检测：因子计算中是否用了未来才可得的字段？
- 字段映射是否正确？(`valuation.market_cap` 等映射关系是否匹配 spec)
- 异常值处理是否按 spec 执行？
- 中性化处理是否正确？
- 换仓逻辑是否有系统性偏向？
- 退市股票是否被纳入处理？(或被静默剔除？)

**结果：** 审核通过 → 进入 Step 4；审核不通过 → 退回 Flash 修正，Step 2→3 循环

---

## 第四步：结果分析 & 沉淀（高级模型）

**输入：** Step 2 的执行结果 + Step 3 的审核结论 + 执行偏差记录

**分析流程：**

```
1. 确认代码审核通过了 → 结果可信 ✅
   代码审核不通过 → 结果不可信，退回修正
   
2. 按 spec 中预注册的通过标准逐条判定：
   假设：IC_IR > 0.3？
   Q1 多头跑赢基准 2%？
   Q1-Q5 分组单调？
   
3. 如果"通过" → 是否有额外发现？
   按 thinking-prompt 做进一步的归因分析
   更新 research/ 记录

4. 如果"不通过" → 诊断原因：
   假设错误？因子在 A 股不存在定价能力
   数据问题？PIT 模式未正确实现？
   时间区间选择？换一个区间是否翻转？
   股票池选择？仅在特定样本有效？
   产生新的假设 → 进入下一轮 spec 设计

5. 如果"不确定" → 明确不确定性来源：
   是统计不显著还是效应量太小？
   需要更大的样本？更长的回测期？
   生成新的 research-spec
```

**沉淀产出：**
- 更新 `research/` 下的研究报告
- 有效因子 → 记录因子参数和行为特征
- 无效因子 → 记录原因，避免重复尝试
- 更新 spec 库（可以作为后续类似分析的模板）

---

## 第三步补充：交互式澄清通道

**原则上** Flash 不修改 spec，但实践中存在 spec 无法覆盖所有细节的情况。

**允许的例外：** 当 Flash 遇到以下情况时，可通过 **一问一答** 的快速通道与高级模型交互：

- 聚宽 API 字段名与 spec 不匹配
- 数据区间内某些字段不可用
- 单调性检验的分档方式不明确
- 参数扫描范围的具体步长

**禁止一问一答的：** 所有涉及假设修改、通过标准调整、因子定义的变更。这些必须走完整的 spec 修订流程。

**交互方式：** Flash → 提出一个具体问题 + 建议的两个方案 → 高级模型回复选哪个。单次对话，不展开讨论。

## 项目全生命周期工作流

```
Phase 1: 单因子验证
  ┌─────────────────────────────────────────────────────────┐
  │ 高级 → research-spec（含预注册通过标准）                   │
  │  Flash → 执行代码 + 结果 + 偏差记录                       │
  │ 高级 → 代码审核（看结果前）                               │
  │ 高级 → 分析结果，对照预注册标准逐条判定                     │
  │  → 通过 → 进入下一个因子                                 │
  │  → 不通过 → 记录原因，放弃或生成新的 spec                   │
  └─────────────────────────────────────────────────────────┘

Phase 2: 多因子合成
  ┌─────────────────────────────────────────────────────────┐
  │ 高级 → 读取 Phase 1 的结构化输出（research/ 下的报告）     │
  │  → 基于因子相关性和独立贡献设计合成方案                    │
  │  → research-spec（含权重方案、参数范围、通过标准）         │
  │  Flash → 执行回测                                        │
  │ 高级 → 代码审核                                          │
  │ 高级 → 归因分析 + 过拟合检验                              │
  │  → 通过 → Phase 3                                       │
  │  → 不通过 → 返回修改合成方案                              │
  └─────────────────────────────────────────────────────────┘

Phase 3: 压力测试 & 稳健性
  ┌─────────────────────────────────────────────────────────┐
  │ 高级 → 设计压力测试方案                                   │
  │  Flash → 批量跑测试，输出结果集                           │
  │ 高级 → Gate 4: 评估是否通过稳健性检验                     │
  │  → 通过 → 可进入实盘准备                                 │
  └─────────────────────────────────────────────────────────┘
```

**迭代终止条件（防止 data mining）：** 同一个假设连续两轮因子 IC_IR < 0.2 → 放弃该因子方向，进入下一轮前需记录"为什么继续迭代这个方向仍有合理性"，否则停止。

## 高级模型的触发时机

**必须由高级模型处理：**
1. 首次设计任何 research-spec
2. 代码审核（独立于结果分析）
3. 分析回测/IC 结果并做 Gate 判定
4. 决策因子去留、权重方案、参数选择
5. 判断过拟合、制度断点影响、宏观状态依赖
6. 设计下一步迭代方向

**可以直接交给 Flash 的机械性工作：**
1. 按已通过的 spec 编写聚宽代码
2. 执行已确定参数的回测
3. 运行批量参数扫描
4. 整理数据、格式化输出

---

## 抑制确认偏误的设计

本流程中，高级模型同时承担"实验设计"和"结果判定"两个角色。为缓解自我确认循环，设置以下机制：

1. **结果分析前必须完成代码审核** — 避免带着结论偏好去挑代码毛病
2. **通过标准在 spec 中预注册** — 结果出来后不能调整
3. **不通过时强制诊断"假设错误还是实现错误"** — 区分真负和假负
4. **连续两轮不通过后放弃** — 防止无限迭代到"偶然通过"
5. **所有分析结论必须有对应数据支持** — 不能以"根据经验"作为判断依据
6. **research/ 下的历史记录在分析前先读取** — 看过往类似假设的结论，保持一致性

---

## 关键约束

- Flash 不修改 spec，不解释结果，不决定下一步方向
- 每次 handoff 必须有明确的书面产出物
- spec 中预注册的通过标准不可事后调整
- 代码审核是结果分析的前置条件，不可跳过
- 结果不通过时，强制记录"放弃"或"修订后的 spec"，不允许未记录的新尝试
