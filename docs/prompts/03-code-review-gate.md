# Prompt 03 — 代码审核门禁（高级模型，结果分析之前）

> 本文件是**结果分析的前置门禁**。高级模型在看任何回测数字之前，必须先按本
> 清单审核 Flash 交付的代码。审核不通过 → 退回 Flash 修正，Step 2→3 循环；
> 审核通过 → 才可进入 `04-result-adjudication.md` 的结果分析。
>
> 这道门禁的存在是为了防止**带着结论偏好去挑代码毛病**——先看结果的人会不
> 自觉地放大支持结论的代码问题、淡化反对结论的代码问题。

---

## 为什么代码审核必须先于结果分析

人类与模型都有确认偏误：一旦先看到"漂亮"的回测收益，再审代码时会不自觉地为
代码辩护；一旦先看到"难看"的结果，会不自觉地在代码里挑毛病来否定结果。

epic-leek-quant 的工程纪律要求：**代码审核在看结果之前完成，且审核结论必须
落盘**。这是 `docs/agent-workflow.md` 第三步的核心约束，不可跳过。

## 审核清单（逐条打勾，不通过即退回）

### A. PIT 数据语义（对应 C10 / B8）

- [ ] 数据访问是否全部经 `joinquant/data_layer.py`？有无直接 `get_fundamentals`？
- [ ] `get_fundamentals(date=)` 的 date 参数是否为换仓日（季报披露截止后首个交易日）？
- [ ] 是否对关键股票做了 `is_data_available_at` 校验（财报披露日早于换仓日）？
- [ ] 是否处理了财报修正覆盖历史的风险（聚宽 PIT 回溯隐患）？
- [ ] 后复权价格是否正确使用？

### B. 撮合口径（对应 C8 / C9 / B3 / B4）

- [ ] 退市股是否纳入？是否按 `get_delisted_calendar` 取实际退市回报？
- [ ] 涨跌停是否经 `simulate_limit_order` 撮合？有无"跳过"简化？
- [ ] 调仓顺序是否"先卖后买"（`rebalance_ordered`）？
- [ ] 停牌期间是否冻结？复牌首日是否按实际成交价计入？
- [ ] T+1 是否遵守？

### C. 因子计算正确性（对应 B1 / B2）

- [ ] `debt_to_assets` 是否"先算比率再取中位数"（B1 修复）？
- [ ] 二值因子是否直接等权求和，**未**做 Z-score（B2 修复）？
- [ ] 连续因子是否做了 1%/99% winsorize + Z-score？
- [ ] 字段映射是否与 spec 一致（`valuation.market_cap` 等）？
- [ ] 国企判定是否用 `STATE_OWNERS` 枚举，而非名称模糊匹配？

### D. 数据处理与中性化（对应 C11）

- [ ] 异常值处理是否按 spec（剔除净利润为负、市值过小、资产负债率>100%）？
- [ ] 缺失值处理是否按 spec（关键字段缺失即剔除）？
- [ ] 是否同时给出行业中性化 IC 与市值中性化 IC？
- [ ] 市值加权 IC 对照组是否产出？

### E. 成本模型（对应 C2–C6 / B7）

- [ ] 是否经 `apply_cost_model`（万 2.5 / 千 1 / 5 元 / 0.3%）？
- [ ] 有无自行 `set_commission` 覆盖统一口径？
- [ ] 印花税是否仅卖出（不是揉进卖出佣金）？
- [ ] 最低佣金 5 元在等权小单下的影响是否校准？

### F. 换仓频率与基准（对应 C1 / C7 / B9）

- [ ] 是否季度调仓（对齐披露截止日）？月频仅作敏感性对照？
- [ ] 基准是否主沪深 300 全收益 + 辅中证 800 全收益？
- [ ] 是否报告跟踪误差 TE 与信息比率 IR（而非仅超额绝对值）？

### G. ST 与次新股（对应 B5 / B6）

- [ ] ST 判定是否用 `get_extras('is_st', ...)`，**未**用 `display_name.startswith`？
- [ ] 次新股剔除是否用 `is_new_stock`（已补 import）？

### H. 输出完整性

- [ ] spec.outputs 要求的表格是否齐全（IC 描述统计、Q1-Q5 分组、分市值档 IC、
      分行业 IC、市值中性化 IC）？
- [ ] Q1 多头绝对收益是否单独报告（而非仅 Q1-Q5 多空）？
- [ ] 净值曲线 CSV 与月度 IC 序列 CSV 是否产出？
- [ ] 执行偏差记录是否落盘？

### I. 并行执行合规（仅当 spec 标注 parallel.enabled=true 时检查）

- [ ] 各子代理是否**只读**共享 data_layer，无自建数据缓存、无绕过？
- [ ] 每个 spec_id 是否各有独立偏差记录（**未合并**）？
- [ ] 全部切片结果是否全量进 `research/_index.md`（含失败、含 IC_IR<0.2 的）？
      缺失即视为择优上报嫌疑，退回补齐。
- [ ] spec 要求的 DSR 多重比较修正是否计算并报告？
- [ ] 审核本身是否仍**逐份串行**做（未因并行而批量过）？

## 审核结论模板

落盘到 `research/reports/<spec_id>-code-review.md`：

```markdown

## 审核日期：<ISO>
## 审核人：高级模型
## 审核时是否已看结果：<必须为 否>

## 逐项结论
| 类别 | 项 | 通过? | 备注 |
|------|----|------|------|
| A. PIT | A1 data_layer | ✅ | |
| A. PIT | A2 换仓日 | ✅ | |
| ... | ... | ... | ... |

## 发现的问题
1. [P0] <问题> → 退回 Flash 修复
2. [P1] <问题> → 退回 Flash 修复

## 总结论
- [ ] 通过 → 进入 04-result-adjudication
- [ ] 不通过 → 退回 Flash，列出修复项

## 备注
<对后续结果分析值得注意的实现细节>
```

## 关键纪律

- **审核时不得查看 `results/<spec_id>/` 下的任何数字**。只看代码与偏差记录。
- 审核结论必须落盘，且"是否已看结果"字段必须为"否"。
- P0 问题（影响结论可信度）必须退回修复，不得"先通过再补"。
- P1 问题（影响严谨性）原则上退回，除非 spec 明确允许 Phase 内修复。
